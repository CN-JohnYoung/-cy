# 1. 卡牌

## 1.1 卡牌模板

卡牌模板用于描述卡牌的设计数据。游戏中通常不会直接修改模板，模板应当作为只读的数据源使用。

卡牌模板记录卡牌的默认属性、默认单位、是否可以三连，以及卡牌拥有的入口效果模板 id。模板不记录卡牌所在区域、归属玩家、冻结状态、三连合并状态、升级结果、slot 状态，也不记录商店阶段或战斗阶段中的运行期变化。

卡牌模板中的效果数组通常只保存入口效果 id，尤其是主动效果 id。入口效果链接的诱发效果、复原效果等辅助效果不需要直接写在卡牌模板里，而是在卡牌实例初始化时根据效果模板的链接表展开。

## 1.2 卡牌实例

卡牌实例是本局游戏中的实体。游戏初始化时，系统会依据卡牌模板和卡池配置批量创建常驻卡牌实例。临时卡牌大多数情况下也依据模板创建，只是会标记为临时卡牌。

实例会继承模板数据，但之后所有商店阶段、战斗阶段和效果系统产生的变化都写在实例上，包括 zone、owner、freezeState、单位变化、追加或移除效果、三连合并、临时状态等。

玩家角色同样是一种特殊的卡牌实例。玩家的商店等级、当前晶体矿、当前晶体矿上限、晶体矿最大上限、当前瓦斯、升本费用等资源与商店参数，都记录在玩家角色上。基础操作的 cost 与收益通过读取和修改玩家角色上的特殊单位或对应数值完成。

## 1.3 按模板初始化实例

卡牌实例应有一个统一的“按模板初始化实例”流程。该流程接收 cardInstanceId、cardTemplateId、目标 zone、owner、是否临时等参数，负责把模板默认数据写入实例：基础属性、单位、入口效果展开后的挂载记录、slot 分配，以及运行期状态的默认值。

该流程用于三种场景：

1. 游戏初始化时创建常驻卡池。
2. 游戏中创建临时卡牌。
3. 常驻卡牌返回卡池时重置状态。

常驻卡牌返回卡池时，不保留购买、升级、冻结、slot、临时效果等运行期状态，而是对该实例重新执行“按模板初始化实例”的流程。返池和初始化共用同一个模板拷贝规则，区别在于返池会复用已有 instanceId/templateId，并将卡牌放回卡池。

## 1.4 常驻卡与临时卡

游戏初始化时创建的常驻卡牌实例初始位于卡池 zone，无 owner，isTemporary 为 false，并且已经拥有由模板入口效果展开出的基础效果挂载记录与 slot。商店刷新只从卡池 zone 的常驻实例中抽取，并移动到玩家商店区。

临时卡牌通常也根据模板创建，但 isTemporary 为 true。临时卡离开商店、手牌、场上或被刷新处理时，不返回常驻卡池；应当释放其 slot 与区域引用，并将实例设为失活或进入临时废弃状态。若未来需要让临时卡继续被某些效果追踪，可以保留 inactive 实例记录，但默认不参与卡池抽取。

## 1.5 单位与场上上限

普通单位与特殊单位的函数通常分开写。

每个玩家场上所有卡牌的普通单位总和有上限，记录在全局常量里。给商店、卡池、手牌里的卡牌添加单位通常不受该上限限制。只有卡牌在场上获得普通单位时，才走受上限限制的场上添加函数，并更新玩家场上普通单位总量缓存。

卡牌入场时如果总单位数量超过上限，可以暂时溢出，但是后续场上的卡牌无法再获得普通单位。卡牌离开场上、返池、出售或重置时，需要同步扣减或重算该玩家场上普通单位缓存，避免缓存与实例内容漂移。

## 1.6 升级与追加内容

给卡牌添加升级属于玩家角色的主动效果。升级的目标通常是某张卡牌，升级内容可以是单位、战斗阶段 buff、诱发效果、主动效果或其他可被卡牌实例记录的内容。

升级不应当作为独立于效果系统之外的特殊流程，而应当通过玩家角色主动效果触发，经过尝试 event 联动后，在 operation 中给目标卡牌添加对应内容。

升级或临时追加效果不直接写入模板。它们以运行期挂载记录加入目标卡牌实例：为入口升级效果分配新 slot，展开它的一层链接效果，并将组内记录追加到 lv_effectIds/lv_slotIds。若追加导致 gc_CardEffectMax 超限，则追加失败。卡牌返池或升级被移除时，这些追加挂载记录与对应 slot 一起释放，不影响模板。

# 2. 效果

## 2.1 效果模板

效果模板用于描述效果的设计数据。监听到对应 event 时，效果会尝试判断 condition 是否满足；如果满足，则 selectTarget 选取目标，并尝试消耗 cost；cost 支付成功后，对每一个 target 执行 operation。如果 condition 不满足或 cost 无法支付，则该效果中断，不继续执行 operation。

效果模板记录效果类型、监听事件、condition/cost/target/operation 函数 id、参数，以及一层 linkedEffectTemplateIds。被链接的效果不允许继续链接其他效果；设计上应当避免递归与循环。

## 2.2 效果实例

游戏初始化时，每个 EffectTemplate 只生成一个对应的 EffectInstance。因此效果模板与效果实例是一一对应关系；实现上可以保持 effectTemplateId == effectInstanceId，但语义上仍区分模板 id 与实例 id。

EffectInstance 不保存 sourceCardInstanceId。它代表可复用的效果逻辑实例，同一个效果实例可以被多张卡、甚至同一张卡的多个效果组引用。不同卡牌或不同效果组之间的状态差异由 SlotInstance 维护。

## 2.3 效果挂载记录

CardInstance.lv_effectIds[i] 与 CardInstance.lv_slotIds[i] 一一对应。卡牌实际拥有的效果不是单纯的 effect id，而是每一条 (effectInstanceId, slotInstanceId) 挂载记录。

同一张卡允许出现重复 effectInstanceId。如果两个入口效果链接了同一个辅助效果，该辅助效果会作为两条记录出现，并分别绑定各自效果组的 slot。查询机遍历时按挂载记录触发，不按 effect id 去重。

gc_CardEffectMax 限制的是展开后的真实挂载记录总数，包括入口效果、链接效果、升级或临时追加效果。模板设计或运行期追加如果导致挂载记录超过上限，应当失败或视为设计错误。

## 2.4 链接效果与共享 Slot

卡牌模板通常只保存入口效果 id。卡牌实例按模板初始化时，会对每个入口效果执行一次效果组展开：

1. 将入口效果加入 CardInstance.lv_effectIds。
2. 为这个入口效果分配一个新的 SlotInstance。
3. 读取该效果模板的 linkedEffectTemplateIds。
4. 将链接的诱发效果、复原效果等辅助效果也加入同一张卡牌实例。
5. 链接效果对应的 lv_slotIds 使用入口效果同一个 slot。

入口效果和它展开出的链接效果组成一个效果组。组内所有挂载记录共享同一个 slot，因此主动效果写入的状态可以被它链接出来的诱发效果或复原效果读取。

效果链接只允许一层。linkedEffectTemplateIds 只应出现在入口效果上，被链接效果必须保持 linkedEffectTemplateIds 为空。初始化卡牌效果时不递归展开，避免循环、重复挂载和 slot 归属歧义。

## 2.5 Slot 实例

SlotInstance 记录“某张卡上的某个效果组”的状态参数。它应当记录该 slot 属于哪个卡牌实例、哪个效果实例或效果组、哪个玩家，以及状态参数。

每当一个入口效果组被挂载到卡牌上时，都会分配一个新的 slot。卡牌重置、返池、升级移除或临时卡失活时，应释放或清空对应 slot，并解除卡牌上的挂载记录。

# 3. 商店阶段

## 3.1 区域与双向追踪

通常来说，卡牌处于场上、卡池、商店、手牌、被合并等区域。商店阶段维护双向追踪机制：卡牌实例自己通过 lv_ZoneId 和 lv_ownerPlayerId 记录所在区域与归属玩家；玩家手牌、场上、商店等数组只保存卡牌实例 id，用于快速遍历、显示和定位槽位。

卡牌实例数组中包含本局游戏内所有卡牌。卡牌的位置移动与状态改变通常不会创建新的实例，而是修改卡牌实例的 zone、state、冻结状态等参数。刷新商店、购买卡牌、出售卡牌、手牌入场等流程都应当维护卡牌自身状态与玩家区域数组的一致性。

所有移动都应走统一的区域移动函数，而不是分散手写字段修改。购买、手牌入场、出售、刷新旧商店陈列、冻结/解冻，都必须维护三类状态一致：卡牌实例字段、玩家区域数组、slot/效果活跃状态。

## 3.2 商店阶段的事件驱动模型

商店阶段中，卡牌、玩家角色与全局特效都由 event 驱动。玩家操作不会直接绕过效果系统修改状态，而是发出对应 event，随后由查询机按顺序遍历所有活跃效果。

当查询机发现某个效果监听当前 event 时，执行流程为：

1. 判断 condition 是否满足。
2. selectTarget 选取目标。
3. 尝试支付 cost。
4. 对目标执行 operation。

如果 condition 不满足或 cost 无法支付，则该效果中断，不继续执行 operation。operation 执行过程中可以发出新的 event，诱发后续效果响应。

通常情况下，玩家 A 的操作只影响玩家 A 自己的商店界面与相关卡牌。其他玩家默认不会响应玩家 A 的商店操作。卡池里的卡牌通常不处于活跃状态，不进入查询机遍历。

活跃效果的默认大类遍历顺序为：

1. 全局特效。
2. 玩家角色特效。
3. 场上卡牌特效。
4. 手牌特效。
5. 商店特效。

同一类别内部的顺序不在本节强制约束，后续可以按照槽位顺序、效果挂载顺序或创建顺序进一步约定。多个效果修改同一个操作上下文时，按照监听顺序依次叠加修改，后执行的效果读取前面效果已经修改后的值。

## 3.3 主动效果、诱发效果与基础操作

目前 effect 主要分为主动效果与诱发效果，未来可以扩展诱发可选发动效果。

玩家基础操作本质上也是主动效果。提升商店等级、刷新商店、冻结商店、购买卡牌、出售卡牌、手牌入场、添加升级、发动卡牌主动效果、发动角色主动效果、发动全局主动效果，都应当通过主动效果流程描述。

商店阶段的基础操作包括：

1. 提升商店等级。
2. 刷新商店。
3. 冻结商店。
4. 购买卡牌到场上。
5. 购买卡牌到手牌。
6. 将手牌入场。
7. 将手牌出售。
8. 将场上卡牌出售。
9. 给卡牌添加升级。
10. 发动卡牌主动效果。
11. 发动角色主动效果。
12. 发动全局主动效果。

这些基础操作默认视为玩家角色的主动效果。这样角色效果、全局效果、卡牌效果都可以通过监听 event 影响购买费用、出售收益、刷新规则、升级内容等结算参数。

## 3.4 尝试事件与完成事件

基础操作正式结算前，会先发出尝试类 event，例如尝试购买卡牌、尝试出售卡牌、尝试刷新商店、尝试提升商店等级等。

尝试类 event 只在玩家基础操作进行前发出，用于让其他活跃效果修改本次操作上下文。监听尝试类 event 的效果可以改变 condition、cost、target、收益、费用、是否允许执行等临时结算参数。

尝试类 event 结算完后，基础操作读取被修改后的上下文继续执行。如果修改后的 condition 无法通过，或 cost 无法支付，则基础操作失败。

基础操作失败时：

1. 不执行实际 operation。
2. 不发出购买完成、出售完成等完成类 event。
3. 不发出购买失败、出售失败等失败 event。
4. 只向当前玩家弹出失败原因提示。

基础操作成功结算后，再发出完成类 event，例如购买完成、出售完成、刷新完成等，用于触发“每当你购买一张牌后”之类的诱发效果。

卡牌与卡牌之间的普通互动通常不使用尝试类 event。普通效果是在自己的 operation 结算完成后，再发出新的结果类 event 诱发其他效果。

## 3.5 购买、入场与出售

购买卡牌到场上与购买卡牌到手牌属于优先级最高的基础操作。玩家操作一旦被接受，就会立刻进入结算流程，不会被其他效果插入导致目标区域空间变化。

购买前只需要检查目标区域是否有空位：

1. 购买到场上时，检查场上是否有空位。
2. 购买到手牌时，检查手牌是否有空位。

购买成功后，将卡牌实例从商店区移动到目标区域，修改卡牌 zone，并同步更新玩家商店数组、手牌数组或场上数组。商店数组对应槽位清空，卡牌移动到手牌或场上目标槽位，并更新 owner/zone。购买完成后发出对应完成类 event。

手牌入场时，将手牌中的卡牌移动到场上，并同步维护 zone 与数组。手牌出售与场上出售都属于玩家角色主动效果，可以产生固定收益，也可以被尝试出售类 event 修改收益或限制出售。

出售时从当前区域数组移除。常驻卡按模板重置后回卡池，临时卡释放 slot 与区域引用后失活。

## 3.6 商店刷新与卡池抽取

卡池不是玩家区域数组，而是由 CardInstance 的 zone 表示。刷新商店时，只从 zone 为卡池、等级不高于玩家当前酒馆等级、且满足抽取条件的常驻卡牌实例中抽取候选卡。已经位于任意玩家商店、手牌、场上、合并区、临时废弃区等区域的卡牌，不会被抽到。

刷新候选规则默认为：卡牌星级不高于玩家当前酒馆等级，并按照相同权重抽取。抽中的卡牌实例从卡池移动到当前玩家商店区，并将实例 id 写入玩家商店数组。

刷新旧商店陈列时，需要根据卡牌实例的 isTemporary 判断是否返回卡池。非临时卡牌可以返回卡池，后续仍可能再次出现在商店；返回卡池时按模板重新初始化实例状态。临时卡牌不返回卡池，也不会再次出现在商店，例如被变形成的卡牌、凭空创造出的卡牌等。

刷新 effect 需要读取商店中每张卡牌的冻结状态，并根据冻结类型分类处理：是否保留、是否解除冻结、是否返回卡池、是否销毁或失活、是否释放槽位并补充新卡。刷新后必须保证商店数组、卡牌 zone、owner、冻结状态一致。

## 3.7 冻结机制

冻结状态应当作为卡牌实例上的 int 变量记录，而不是简单 bool。该变量可以命名为 lv_freezeStateId。0 表示未冻结，普通冻结与特殊冻结使用不同状态值区分。

玩家基础操作“冻结商店”会将当前商店陈列中的卡牌设置为普通冻结状态。下一个回合进入商店阶段时，被冻结的卡牌根据冻结状态保留在商店陈列中；没被冻结的卡牌或空位则按刷新规则处理。

特殊冻结通常由卡牌、角色或全局效果产生，拥有自己的取消规则。回合开始、刷新商店、取消冻结、购买卡牌、卡牌离开商店等操作都应当读取冻结状态，并在对应 effect 中分类处理。

## 3.8 商店阶段结束

商店阶段通常有倒计时。倒计时结束后，系统中断玩家后续操作输入，并发出商店阶段结束 event，随后进入战斗阶段。

按照银河编辑器的触发器执行方式，玩家操作一旦被接受，就会立刻执行，其优先级高于时间倒计时。因此倒计时不会强制截断已经开始执行的 effect 链。当前操作及其同步触发的 event 链结算完成后，再发出商店阶段结束 event 并进入战斗阶段。

## 3.9 错误处理与验证重点

模板层错误应当视为设计错误，尽量在初始化或专门检查函数中暴露，而不是游戏中静默处理。包括：入口效果展开后超过 gc_CardEffectMax、被链接效果继续链接其他效果、模板引用不存在的效果模板、单位类型超出普通/特殊单位区间、卡牌模板 id 或效果模板 id 无效等。

运行期失败是正常游戏结果，应当只影响本次操作，不破坏状态。例如购买时目标区域满、刷新时卡池候选不足、升级追加效果超出卡牌效果槽位、费用不足、condition 不满足等。

建议形成几类底层原子操作：按模板初始化或重置卡牌实例、挂载一个效果组、释放某张卡的所有 slot、移动卡牌区域、常驻卡返池、临时卡失活、场上单位增减。商店阶段的购买、出售、刷新、入场等高级流程只组合这些原子操作。

验证重点应覆盖几个核心不变量：模板初始化后实例字段与模板一致；入口效果能展开一层链接效果并共享 slot；重复 effect id 在同一卡上可用不同 slot 区分；返池会清理升级、冻结、slot 并恢复模板状态；临时卡不会回到常驻卡池；区域数组与实例 zone/owner 始终一致；场上单位缓存与场上卡牌实际普通单位总量一致。
