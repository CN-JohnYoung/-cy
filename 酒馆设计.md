# 0. 总体设计
当前实现中，数组下标应当与结构体自带 id 保持一致。
结构体自带 id 不是主要寻址方式，而是冗余校验与未来扩展字段。
初始化或数据校验阶段应检查二者一致。

# 1. 卡牌

## 1.1 卡牌模板

卡牌模板用于描述卡牌的设计数据。绝大多数玩家操作与普通效果不会直接修改模板，模板默认作为实例初始化与重置时读取的源数据使用。未来可以存在少数特殊效果修改模板；模板被修改后，默认只影响之后按模板初始化或重置出来的卡牌实例，不自动同步已经存在的 CardInstance。

卡牌模板记录卡牌的默认属性、默认单位、是否可以三连，以及卡牌拥有的卡牌效果模板 id。模板不记录卡牌所在区域、归属玩家、冻结状态、三连合并状态、升级结果、slot 状态，也不记录商店阶段或战斗阶段中的运行期变化。

卡牌模板中的效果数组保存 CardEffectTemplate id。一个 CardEffectTemplate 可以把主动效果、诱发效果、复原效果等多个 EffectInstance 组装为同一个卡牌特效组，并在卡牌实例初始化时挂载为 CardEffectInstance。

CardTemplate 应记录 templateVersion。特殊效果修改模板的默认属性、默认单位或默认效果配置时，必须递增该版本号，用于判断已有实例是否仍然等同于当前模板派生结果。

如果某个效果需要让模板改动影响已有卡牌，必须显式执行额外操作：遍历引用该 templateId 的 CardInstance，按 zone、owner、state 等条件筛选目标，再对目标实例执行同步、增量修改或重置，并按需要重建 CardEffectInstance 与 SlotInstance。

## 1.2 卡牌实例

卡牌实例是本局游戏中的实体。游戏初始化时，系统会依据卡牌模板和卡池配置批量创建常驻卡牌实例。临时卡牌大多数情况下也依据模板创建，只是会标记为临时卡牌。

实例会继承模板数据，但之后所有商店阶段、战斗阶段和效果系统产生的变化都写在实例上，包括 zone、owner、freezeState、单位变化、追加或移除效果、三连合并、临时状态等。

普通玩家操作应当面向 CardInstance 结算，而不是直接操作 CardTemplate。改模板表示修改之后初始化或重置时使用的默认源数据；改实例表示修改当前局面中已经存在的实体。

CardInstance 应记录 templateVersion 与 isTemplateDirty。templateVersion 表示该实例上次按模板初始化或重置时读取到的模板版本；isTemplateDirty 表示实例的模板派生内容是否在运行期被改动过。单位变化、升级、追加或移除效果、变形、三连合并等会使 isTemplateDirty 变为 true；单纯移动 zone/owner、普通冻结状态变化不需要设置该标记，因为这些字段本来就不是模板派生内容。

玩家角色同样是一种特殊的卡牌实例。玩家的商店等级、当前晶体矿、当前晶体矿上限、晶体矿最大上限、当前瓦斯、升本费用等资源与商店参数，都记录在玩家角色上。基础操作的 cost 与收益通过读取和修改玩家角色上的特殊单位或对应数值完成。

## 1.3 按模板初始化实例

卡牌实例应有一个统一的“按模板初始化实例”流程。该流程接收 cardInstanceId、cardTemplateId、目标 zone、owner、是否临时等参数，负责把模板默认数据写入实例：基础属性、单位、入口效果展开后的挂载记录、slot 分配，以及运行期状态的默认值。

该流程用于三种场景：

1. 游戏初始化时创建常驻卡池。
2. 游戏中创建临时卡牌。
3. 常驻卡牌返回卡池时重置状态。

常驻卡牌返回卡池时，不保留购买、升级、冻结、slot、临时效果等运行期状态，而是对该实例重新执行“按模板初始化实例”的流程。返池和初始化共用同一个模板拷贝规则。通常返池会复用已有 instanceId/templateId，并将卡牌放回卡池；变形导致的常驻身份补回规则见 1.7。

返池重置读取的是当前 CardTemplate，而不是游戏开始时的原始模板备份。如果本局内有特殊效果修改过模板，之后新建实例、临时卡创建和常驻卡返池重置都会读取修改后的模板。

为了避免大量商店刷新时反复执行完整结构体拷贝，返池流程可以先检查 CardInstance.isTemplateDirty 与模板版本：如果 isTemplateDirty 为 false，且 CardInstance.templateVersion 等于当前 CardTemplate.templateVersion，则该实例的模板派生内容仍与当前模板一致，返池时只需要更新 zone、owner、freezeState 等运行期字段；否则必须按当前模板完整重置实例，重建单位、基础效果挂载与 slot，并将 isTemplateDirty 设回 false，把实例 templateVersion 更新为当前模板版本。

## 1.4 常驻卡与临时卡

游戏初始化时创建的常驻卡牌实例初始位于卡池 zone，无 owner，isTemporary 为 false，并且已经拥有由模板入口效果展开出的基础效果挂载记录与 slot。商店刷新只从卡池 zone 的常驻实例中抽取，并移动到玩家商店区。

临时卡牌通常也根据模板创建，但 isTemporary 为 true。临时卡离开商店、手牌、场上或被刷新处理时，不返回常驻卡池；应当释放其 slot 与区域引用，并将实例设为失活或进入临时废弃状态。若未来需要让临时卡继续被某些效果追踪，可以保留 inactive 实例记录，但默认不参与卡池抽取。

常驻身份不等同于某个 instanceId 永久不变。若常驻卡被变形，当前实例会转为临时卡；常驻卡池名额由额外创建并返回卡池的新实例承接。

## 1.5 单位与场上上限

普通单位与特殊单位的函数通常分开写。

每个玩家场上所有卡牌的普通单位总和有上限，记录在全局常量里。给商店、卡池、手牌里的卡牌添加单位通常不受该上限限制。只有卡牌在场上获得普通单位时，才走受上限限制的场上添加函数，并更新玩家场上普通单位总量缓存。

卡牌入场时如果总单位数量超过上限，可以暂时溢出，但是后续场上的卡牌无法再获得普通单位。卡牌离开场上、返池、出售或重置时，需要同步扣减或重算该玩家场上普通单位缓存，避免缓存与实例内容漂移。

## 1.6 升级与追加内容

给卡牌添加升级属于玩家角色的主动效果。升级的目标通常是某张卡牌，升级内容可以是单位、战斗阶段 buff、诱发效果、主动效果或其他可被卡牌实例记录的内容。

升级不应当作为独立于效果系统之外的特殊流程，而应当通过玩家角色主动效果触发，经过尝试 event 联动后，在 operation 中给目标卡牌添加对应内容。

升级或临时追加效果不直接写入模板。它们以运行期 CardEffectInstance 加入目标卡牌实例：根据 CardEffectTemplate 中的 logicSlotId 分配真实 slot，并将该卡牌效果实例追加到卡牌实例的挂载列表。若追加导致 gc_CardEffectMax 超限，则追加失败。卡牌返池或升级被移除时，这些追加挂载记录与对应 slot 一起释放，不影响模板。

## 1.7 变形

变形是在当前 CardInstance 上改写信息，而不是直接替换为另一个已有实例。变形时保留当前 instanceId、当前位置和区域引用，将 lv_templateId 改为目标模板 id，释放当前旧 CardEffectInstance 与 SlotInstance，并按目标 CardTemplate 重写卡牌名称、等级、阵营、单位、基础效果挂载、SlotInstance 等模板派生内容。

变形前如果当前实例是常驻卡，应先记录原 templateId，然后额外创建一个按原模板初始化的新常驻实例并放入卡池。当前被变形的实例改为 isTemporary = true，之后离开商店、手牌、场上或被刷新处理时按临时卡规则释放或失活，不再返回常驻卡池。

变形前如果当前实例已经是临时卡，则只改写当前实例，不额外创建返池实例。临时卡再次变形不会向常驻卡池补入任何卡牌。

# 2. 效果

## 2.1 效果模板

EffectTemplate 用于描述一类可复用的效果逻辑，只记录 event -> condition -> target -> cost -> operation 的执行流程。

EffectTemplate 不记录具体参数，不记录 activeZoneId，不记录 activeStateId，也不记录 link 关系。它更接近一个效果函数定义。例如“回合结束时，如果本卡牌具有 X，则获得 N 个 Y”是一条 EffectTemplate；其中 X、N、Y 的具体值不属于模板。

监听到对应 event 时，查询机会尝试判断 condition 是否满足；如果满足，则 selectTarget 选取目标，并尝试消耗 cost；cost 支付成功后，对每一个 target 执行 operation。如果 condition 不满足或 cost 无法支付，则该效果中断，不继续执行 operation。

## 2.2 效果实例

EffectInstance 是带具体参数的效果定义。它引用一个 EffectTemplate，并为该模板需要的参数提供具体数值。

例如 EffectTemplate 表示“回合结束时，如果本卡牌具有 X，则获得 N 个 Y”，某个 EffectInstance 可以把 X 设为反应堆，N 设为 1，Y 设为陆战队员。另一个 EffectInstance 可以复用同一个 EffectTemplate，但填入不同的 X、N、Y。

EffectInstance 的参数按类型分为整数参数和字符串参数。数值类参数写入 intParamArray，单位 id、挂件 id、事件 id 等文本类参数写入 stringParamArray，避免在效果查询和结算过程中频繁做字符串与整数的类型转换。

EffectInstance 不记录卡牌来源，不记录 zone/state，不记录它和其他效果的 link。它只负责把通用效果逻辑参数化，使同一个 EffectTemplate 可以分发给不同卡牌和不同卡牌特效使用。

## 2.3 卡牌效果模板

CardEffectTemplate 用于描述一张卡牌上的一个卡牌特效组。它把多个 EffectInstance 组装在一起，并负责这些 EffectInstance 的 link、activeZoneId、activeStateId 与逻辑 slot 分组。

CardEffectTemplate 内部使用并行数组描述组内效果：

1. effectInstanceIds[i] 表示第 i 个组内效果使用哪个 EffectInstance。
2. logicSlotIds[i] 表示第 i 个组内效果使用哪个逻辑 slot。
3. activeZoneIds[i] 表示第 i 个组内效果在哪个区域生效。
4. activeStateIds[i] 表示第 i 个组内效果在哪个卡牌状态生效。

这些数组必须一一对应。下标 0 不作为正式数据使用；effectInstanceId、logicSlotId、activeZoneId、activeStateId 中的 0 都表示非法值，不表示“任意”。即便普通状态和金色状态的效果逻辑完全相同，也应分别配置两条 activeStateId 不同的组内效果。

CardEffectTemplate 的 link 关系不放在 EffectTemplate 或 EffectInstance 层。主动效果和诱发效果可能只是共享 slot 数据，并不一定存在通用逻辑上的从属关系，因此它们是否组合在一起应由具体卡牌特效模板决定。

CardEffectTemplate 可以记录 macroId，作为 UI 宏展示标记。macroId 为 0 表示不使用宏展示；非 0 表示这组卡牌效果在 UI 生成时按对应宏规则压缩显示，例如“反应堆：XXX”可以显示为关键词徽章。macroId 不改变 effect 的结算逻辑，查询机仍然只按展开后的 EffectInstance、activeZoneId、activeStateId、slot 等普通挂载记录执行。

宏展示标记不负责自动生成 event、condition、target、cost、operation。底层效果仍需完整配置，macroId 只告诉 UI 如何把这组已配置好的规则展示给玩家。详情页或悬浮提示可以根据 macroId 展开完整说明，例如“回合结束时，如果本卡牌具有反应堆，则 XXX”。

## 2.4 卡牌效果实例与挂载记录

CardEffectInstance 是 CardEffectTemplate 挂载到某张 CardInstance 后的运行期实例。它负责根据 CardEffectTemplate.logicSlotIds 分配真实 SlotInstance，并生成可供查询机遍历的效果挂载记录。

同一个 CardEffectTemplate 内，相同 logicSlotId 对应同一个真实 SlotInstance；不同 logicSlotId 对应不同真实 SlotInstance。logicSlotId 只是模板内的逻辑标识，不是 SlotInstanceId。

例如某个 CardEffectTemplate 中：

1. effectInstanceIds[1] 是主动效果 A，logicSlotIds[1] 为 1。
2. effectInstanceIds[2] 是辅助诱发效果 B，logicSlotIds[2] 为 1。
3. effectInstanceIds[3] 是独立诱发效果 C，logicSlotIds[3] 为 2。

挂载到卡牌实例后，A 与 B 会共享同一个真实 SlotInstance，C 会分配另一个真实 SlotInstance。

卡牌实际拥有的效果不是单纯的 EffectInstanceId，而是每一条由 CardEffectInstance 展开的挂载记录。查询机遍历挂载记录时，先检查事件是否匹配 EffectTemplate.listenEvent，再检查卡牌当前 zone/state 是否匹配该挂载记录来自 CardEffectTemplate 的 activeZoneId/activeStateId，匹配后再进入 condition、target、cost、operation 流程。

查询机进入某条挂载记录的 condition、target、cost、operation 前，应写入当前卡牌效果上下文。该上下文记录当前 effectInstanceId、cardEffectInstanceId、slotInstanceId、ownerCardInstanceId、ownerPlayerId。“本卡牌”“本玩家”“本 slot”都从当前卡牌效果上下文读取，而不是从 event source 读取。

EventContext 表示当前正在处理的事件及其来源。“事件来源卡牌”“事件来源玩家”“事件来源效果”从 EventContext 读取。这样可以区分“本卡牌”和“事件来源卡牌”：例如 B 卡监听 A 卡发出的事件时，B 卡效果中的“本卡牌”是 B，事件来源卡牌是 A。

EventContext 应携带本次事件的目标与结果 payload。单位变化类结果 event 应把实际发生的单位变化写入单位数组，包括 unitId 与实际 unitAmount。监听“其他卡牌获得单位”之类事件的效果，应读取 EventContext 中的目标卡牌与单位数组，而不是回查目标卡牌刚才发生了什么。

单位变化类 event 默认按“某张或某些目标卡牌获得若干种单位”表达。若一次 operation 让多个目标、多个单位类型发生变化，可以登记多条结果 event，或在同一 event 中使用目标数组与单位数组表达；实现时应优先保证 unitAmount 记录实际成功变化的数量，而不是尝试添加的数量。

## 2.5 Slot 实例

SlotInstance 记录“某张卡上的某个卡牌效果逻辑 slot”的运行期状态参数。它应当记录所属卡牌实例、所属玩家，以及整数和字符串状态参数。

普通状态效果和金色状态效果默认拥有各自独立的 logicSlotId 与 SlotInstance。即便它们引用同一个 EffectTemplate 或参数相同的 EffectInstance，也不共享运行期状态，避免三连、复制、追加或组合不同卡牌特效时产生状态混淆。

卡牌重置、返池、升级移除或临时卡失活时，应释放或清空对应 CardEffectInstance 展开的所有 SlotInstance，并解除卡牌上的挂载记录。gc_CardEffectMax 限制的是展开后的真实挂载记录总数，包括普通效果、金色效果、辅助诱发效果、升级或临时追加效果。模板设计或运行期追加如果导致挂载记录超过上限，应当失败或视为设计错误。

# 3. 商店阶段

## 3.1 区域与双向追踪

通常来说，卡牌处于场上、卡池、商店、手牌、被合并等区域。商店阶段维护双向追踪机制：卡牌实例自己通过 lv_ZoneId 和 lv_ownerPlayerId 记录所在区域与归属玩家；玩家手牌、场上、商店等数组只保存卡牌实例 id，用于快速遍历、显示和定位槽位。

卡牌实例数组中包含本局游戏内所有卡牌。卡牌的位置移动与状态改变通常不会创建新的实例，而是修改卡牌实例的 zone、state、冻结状态等参数。刷新商店、购买卡牌、出售卡牌、手牌入场等流程都应当维护卡牌自身状态与玩家区域数组的一致性。

所有移动都应走统一的区域移动函数，而不是分散手写字段修改。购买、手牌入场、出售、刷新旧商店陈列、冻结/解冻，都必须维护三类状态一致：卡牌实例字段、玩家区域数组、slot/效果活跃状态。

## 3.2 商店阶段的事件驱动模型

商店阶段中，卡牌、玩家角色与全局特效都由 event 驱动。玩家操作不会直接绕过效果系统修改状态，而是发出对应 event，随后由查询机按顺序遍历所有活跃效果。

当查询机发现某个效果监听当前 event 时，执行流程为：

1. 判断 condition 是否满足。
2. selectTarget 选取目标。
3. 尝试支付 cost。
4. 对目标执行 operation。

如果 condition 不满足或 cost 无法支付，则该效果中断，不继续执行 operation。operation 不立即递归触发新的 event，而是登记待发 event。当前 event 的所有响应效果都结算完成后，再按登记顺序处理后续 event。

事件结算采用队列式模型。系统任意时刻只处理一个 CurrentEventContext；operation 产生的新 event 写入待处理队列。当前 event 结束后，系统取出队列中的下一个 event 作为新的 CurrentEventContext 继续结算。由于不在 operation 中递归处理新 event，EventContext 不需要压栈，当前卡牌效果上下文也只表示当前正在结算的单条挂载记录。

通常情况下，玩家 A 的操作只影响玩家 A 自己的商店界面与相关卡牌。其他玩家默认不会响应玩家 A 的商店操作。卡池里的卡牌通常不处于活跃状态，不进入查询机遍历。

活跃效果的默认大类遍历顺序为：

1. 全局特效。
2. 玩家角色特效。
3. 场上卡牌特效。
4. 手牌特效。
5. 商店特效。

同一类别内部的顺序不在本节强制约束，后续可以按照槽位顺序、效果挂载顺序或创建顺序进一步约定。多个效果修改同一个操作上下文时，按照监听顺序依次叠加修改，后执行的效果读取前面效果已经修改后的值。

## 3.3 主动效果、诱发效果与基础操作

目前 effect 主要分为主动效果与诱发效果，未来可以扩展诱发可选发动效果。

玩家基础操作本质上也是主动效果。提升商店等级、刷新商店、冻结商店、购买卡牌、出售卡牌、手牌入场、添加升级、发动卡牌主动效果、发动角色主动效果、发动全局主动效果，都应当通过主动效果流程描述。

商店阶段的基础操作包括：

1. 提升商店等级。
2. 刷新商店。
3. 冻结商店。
4. 购买卡牌到场上。
5. 购买卡牌到手牌。
6. 将手牌入场。
7. 将手牌出售。
8. 将场上卡牌出售。
9. 给卡牌添加升级。
10. 发动卡牌主动效果。
11. 发动角色主动效果。
12. 发动全局主动效果。

这些基础操作默认视为玩家角色的主动效果。这样角色效果、全局效果、卡牌效果都可以通过监听 event 影响购买费用、出售收益、刷新规则、升级内容等结算参数。

## 3.4 尝试事件与完成事件

基础操作正式结算前，会先发出尝试类 event，例如尝试购买卡牌、尝试出售卡牌、尝试刷新商店、尝试提升商店等级等。

尝试类 event 只在玩家基础操作进行前发出，用于让其他活跃效果修改本次操作上下文。监听尝试类 event 的效果可以改变 condition、cost、target、收益、费用、是否允许执行等临时结算参数。

尝试类 event 结算完后，基础操作读取被修改后的上下文继续执行。如果修改后的 condition 无法通过，或 cost 无法支付，则基础操作失败。

基础操作失败时：

1. 不执行实际 operation。
2. 不发出购买完成、出售完成等完成类 event。
3. 不发出购买失败、出售失败等失败 event。
4. 只向当前玩家弹出失败原因提示。

基础操作成功结算后，再发出完成类 event，例如购买完成、出售完成、刷新完成等，用于触发“每当你购买一张牌后”之类的诱发效果。

卡牌与卡牌之间的普通互动通常不使用尝试类 event。普通效果是在自己的 operation 结算完成后，再发出新的结果类 event 诱发其他效果。

## 3.5 购买、入场与出售

购买卡牌到场上与购买卡牌到手牌属于优先级最高的基础操作。玩家操作一旦被接受，就会立刻进入结算流程，不会被其他效果插入导致目标区域空间变化。

购买前只需要检查目标区域是否有空位：

1. 购买到场上时，检查场上是否有空位。
2. 购买到手牌时，检查手牌是否有空位。

购买成功后，将卡牌实例从商店区移动到目标区域，修改卡牌 zone，并同步更新玩家商店数组、手牌数组或场上数组。商店数组对应槽位清空，卡牌移动到手牌或场上目标槽位，并更新 owner/zone。购买完成后发出对应完成类 event。

手牌入场时，将手牌中的卡牌移动到场上，并同步维护 zone 与数组。手牌出售与场上出售都属于玩家角色主动效果，可以产生固定收益，也可以被尝试出售类 event 修改收益或限制出售。

出售时从当前区域数组移除。常驻卡按模板重置后回卡池，临时卡释放 slot 与区域引用后失活。

## 3.6 商店刷新与卡池抽取

卡池不是玩家区域数组，而是由 CardInstance 的 zone 表示。刷新商店时，只从 zone 为卡池、等级不高于玩家当前酒馆等级、且满足抽取条件的常驻卡牌实例中抽取候选卡。已经位于任意玩家商店、手牌、场上、合并区、临时废弃区等区域的卡牌，不会被抽到。

刷新候选规则默认为：卡牌星级不高于玩家当前酒馆等级，并按照相同权重抽取。抽中的卡牌实例从卡池移动到当前玩家商店区，并将实例 id 写入玩家商店数组。

刷新旧商店陈列时，需要根据卡牌实例的 isTemporary 判断是否返回卡池。非临时卡牌可以返回卡池，后续仍可能再次出现在商店；返回卡池时按当前模板重新初始化实例状态。临时卡牌不返回卡池，也不会再次出现在商店，例如被变形成的卡牌、凭空创造出的卡牌等。

商店刷新通常也应当把未保留的常驻卡重置到模板。冻结或特殊规则保留在商店中的卡牌没有返池，因此默认不重置；如果某个刷新效果需要“保留但重置”，必须显式执行额外的重置操作。

刷新旧商店陈列时，未保留常驻卡虽然逻辑上执行返池重置，但可以使用模板脏标记与模板版本走快速路径。未被运行期改动且模板版本未变化的常驻卡，只需要从商店区移回卡池并清理 owner/freezeState；被改动过或模板版本落后的常驻卡，才执行完整模板重置。

刷新 effect 需要读取商店中每张卡牌的冻结状态，并根据冻结类型分类处理：是否保留、是否解除冻结、是否返回卡池、是否销毁或失活、是否释放槽位并补充新卡。刷新后必须保证商店数组、卡牌 zone、owner、冻结状态一致。

## 3.7 冻结机制

冻结状态应当作为卡牌实例上的 int 变量记录，而不是简单 bool。该变量可以命名为 lv_freezeStateId。0 表示未冻结，普通冻结与特殊冻结使用不同状态值区分。

玩家基础操作“冻结商店”会将当前商店陈列中的卡牌设置为普通冻结状态。下一个回合进入商店阶段时，被冻结的卡牌根据冻结状态保留在商店陈列中；没被冻结的卡牌或空位则按刷新规则处理。

特殊冻结通常由卡牌、角色或全局效果产生，拥有自己的取消规则。回合开始、刷新商店、取消冻结、购买卡牌、卡牌离开商店等操作都应当读取冻结状态，并在对应 effect 中分类处理。

## 3.8 商店阶段结束

商店阶段通常有倒计时。倒计时结束后，系统中断玩家后续操作输入，并发出商店阶段结束 event，随后进入战斗阶段。

按照银河编辑器的触发器执行方式，玩家操作一旦被接受，就会立刻执行，其优先级高于时间倒计时。因此倒计时不会强制截断已经开始执行的 effect 链。当前操作及其同步触发的 event 链结算完成后，再发出商店阶段结束 event 并进入战斗阶段。

## 3.9 错误处理与验证重点

模板层错误应当视为设计错误，尽量在初始化或专门检查函数中暴露，而不是游戏中静默处理。包括：卡牌效果模板挂载后超过 gc_CardEffectMax、CardEffectTemplate 的并行数组不一致、logicSlotId/activeZoneId/activeStateId 为 0、模板引用不存在的 EffectInstance 或 EffectTemplate、单位类型超出普通/特殊单位区间、卡牌模板 id 或效果模板 id 无效等。

运行期失败是正常游戏结果，应当只影响本次操作，不破坏状态。例如购买时目标区域满、刷新时卡池候选不足、升级追加效果超出卡牌效果槽位、费用不足、condition 不满足等。

建议形成几类底层原子操作：按模板初始化或重置卡牌实例、挂载一个效果组、释放某张卡的所有 slot、移动卡牌区域、常驻卡返池、临时卡失活、场上单位增减。商店阶段的购买、出售、刷新、入场等高级流程只组合这些原子操作。

验证重点应覆盖几个核心不变量：模板初始化后实例字段与模板一致；入口效果能展开一层链接效果并共享 slot；重复 effect id 在同一卡上可用不同 slot 区分；返池会清理升级、冻结、slot 并恢复模板状态；模板脏标记与模板版本能正确区分快速返池和完整重置；临时卡不会回到常驻卡池；区域数组与实例 zone/owner 始终一致；场上单位缓存与场上卡牌实际普通单位总量一致。
